# Audit 2026-05-12

Scope: `dvachbot` bot/site and a light check of `stomchat` logs.  
Mode: live production read-only diagnostics first, minimal isolated code fix after evidence.

## Executive Findings

1. Old bot replies were broken by local cleanup policy, not by Telegram randomness.
2. `PostCopies` was being deleted after 48 hours, while `Posts` stayed. Site history survived; Telegram reply targets died.
3. The bot and site share `post_num`/`Posts`, but bot replies additionally require recipient-specific `PostCopies`.
4. Bot outbound queues are still unbounded, but delivery now prioritizes weekly-active authors before passive readers.
5. The bot process is already heavy for a 630-user bot, but the biggest avoidable memory risk was loading/keeping fanout copy maps in RAM.
6. Shadow mute had a reply leak: `process_shadow_reject()` passed a global post number where `send_message_to_users()` expected a Telegram `message_id`.
7. The site process has bounded WebSocket broadcast queue, but uses in-memory caches and many background workers.
8. The bot now has a rotating runtime telemetry log: `logs/bot_runtime.log`. Before this patch, live console was not reliably mirrored to a current log file.
9. Old-reply fallback quotes were fragmented. They now use one helper and cover text, captions, albums, stickers, voice, video notes, audio, documents, GIFs, and polls.
10. Live queue age is now visible. Completed fanouts log `queue_wait_sec` and `queue_total_sec`; runtime snapshots and `/queues` show queued oldest/current delivery state.
11. Reply-copy health is now visible. Runtime logs and `/queues` show `PostCopies` coverage and whether latest posts are still being recorded.
12. The site process was also memory-heavy. Admin site stats now expose process memory and runtime cache/container sizes.

## Root Cause: Old Replies

Relevant code path:

- `main.py::handle_message()`
- `common/database.py::get_post_info_by_copy()`
- `main.py::process_new_post()`
- `main.py::send_message_to_users()`
- `common/database.py::add_post_copies()`
- `common/database.py::cleanup_old_posts_from_db()`

Previous behavior:

```python
EDITING_LIFETIME = 48 * 3600
DELETE FROM PostCopies WHERE post_num IN (
    SELECT post_num FROM Posts WHERE timestamp < cutoff
)
```

That deletes the only durable map from global `post_num` to per-user Telegram message IDs. After that:

- reply to old post by visible Telegram reply cannot be resolved through `PostCopies`
- bot may parse the post number from visible text
- bot can show `quote_info`
- native Telegram thread reply cannot be made

This is why smaller/other bots appeared better: if their copy map survives longer, replies work farther back.

## Applied Fix

Files changed:

- `common/config.py`
- `common/database.py`
- `main.py`

New configuration:

```python
DB_POST_LIMIT = int(os.getenv("DB_POST_LIMIT", "25000"))
BOT_POST_CACHE_LIMIT = int(os.getenv("BOT_POST_CACHE_LIMIT", "3300"))
BOT_COPY_CACHE_POST_LIMIT = int(os.getenv("BOT_COPY_CACHE_POST_LIMIT", "400"))
POST_COPY_RETENTION_POSTS = int(os.getenv("POST_COPY_RETENTION_POSTS", "12000"))
POST_COPY_RETENTION_DAYS = int(os.getenv("POST_COPY_RETENTION_DAYS", "30"))
BOT_PRIORITY_DELIVERY = enabled by default
BOT_WEEKLY_ACTIVE_DAYS = 7
BOT_WEEKLY_ACTIVE_REFRESH_SEC = 900
```

New behavior:

- `PostCopies` and `ChannelCopies` are retained by age and post distance.
- Startup RAM hydration of copy maps is capped separately by `BOT_COPY_CACHE_POST_LIMIT`.
- Old replies can use indexed SQLite instead of huge RAM maps.
- Shadow reject now sends `reply_to_post` through normal content path and no longer forges `reply_info`.
- Older replies also get a unified `quote_info` fallback when they are more than `QUICK_QUOTE_POST_DISTANCE=330` posts behind the current counter.
- Quote fallback now summarizes attachments from content shapes used by text, media groups, photos, videos, GIFs, documents, audio, voice messages, stickers, video notes, direct `file_id`, image bytes/URLs, and polls.

Verification:

```powershell
python -m py_compile main.py common\database.py common\config.py
```

Result: passed.

Limit:

- Already-deleted historical `PostCopies` cannot be recovered from `Posts`.
- The running bot was restarted after this patch. Future replies improve as new copy rows accumulate under the new retention policy.

## Memory Risk Inventory

Hot maps and queues in `main.py`:

- `messages_storage`
- `post_to_messages`
- `message_to_post`
- `board_data`
- `message_queues`
- `current_media_groups`
- `media_group_timers`
- `pending_edit_tasks`
- `unknown_command_tracker`
- `ROULETTE_EVENTS`
- image/anime usage trackers

Already bounded:

- `recent_messages_cache = deque(maxlen=200)`
- `locally_created_posts = deque(maxlen=500)`
- `sent_media_groups = deque(maxlen=1000)`
- several spam deques have maxlen
- `auto_memory_cleaner()` trims old users and global caches

Risky:

- `message_queues` has `maxsize=0`, meaning unbounded.
- `pending_edit_tasks` must always delete completed tasks; if not, it will leak task objects.
- `current_media_groups` and `media_group_timers` can leak if timer cleanup misses an exception path.
- old reply DB fallback can add one-off `message_to_post` entries without matching `post_to_messages`; cleaner now prunes those orphan reverse-map cache entries.
- `ROULETTE_EVENTS` is static loaded data, not an observed runtime leak, but it is now included in telemetry because config size can still matter.
- site `FastAPICache(InMemoryBackend)` can grow by URL/cardinality until TTL expiration.
- site and bot both use SQLite cache size `-60000`, which can reserve noticeable memory per process.
- site admin APIs now expose process memory and cache/container cardinality through `/api/admin/stats` and `/api/admin/system_health`.
- site cleanup now trims expired FastAPI cache keys, stale thread/url maps, stale flood-tracker keys, expired IP bans, and expired manual troll configs.
- old site image-processing `ProcessPoolExecutor` workers were a concrete OS-level memory leak source; grimdark/thumbnail work now uses bounded `ThreadPoolExecutor` workers and site shutdown calls `shutdown_image_executors()`.

## Queue Delay Diagnosis

Current bot broadcast model:

1. `message_queues[board_id]` receives one item per logical post.
2. One worker per board processes queue items sequentially.
3. `send_message_to_users()` fans out to active recipients in chunks.
4. Chunk size starts at 30 and backs off on FloodWait.
5. CPU saturation from other local projects slows the event loop and all fanout work.

For `/b/`, which has most activity, this is structurally vulnerable. If a post takes long to deliver, every later post waits.

Applied mitigation:

- `common/database.py::get_weekly_active_users()` reads visible authors for the last `BOT_WEEKLY_ACTIVE_DAYS`.
- `main.py::weekly_active_refresh_task()` refreshes that list every `BOT_WEEKLY_ACTIVE_REFRESH_SEC`.
- First refresh runs immediately when the background task starts; there is no intentional 20-second empty-priority phase.
- `send_message_to_users()` orders each recipient list as weekly-active first, passive second.
- A new non-shadow bot post marks its author as hot immediately, so the author does not wait up to 15 minutes for the next refresh.
- Current live sample: `weekly_active_refresh` found `138` priority author-board entries, including `/b/=98`, `/sex/=19`, `/a/=6`.

Remaining future fix:

- persist unsent fanout state in DB for crash-safe resume
- expose queue lag in admin stats:
  - live oldest queued post age is implemented for RAM queue items
  - queue size by board
  - current fanout post
  - active weekly recipients count
  - passive recipients count
  - FloodWait counters
- completed fanout age is now logged as `post_age_sec`

This matches the product reality: active users notice delay; passive users often do not.

## Shadow Mute Review

Implemented shadow restrictions:

- full board shadow mute
- thread-local shadow mute
- media-only shadow bans:
  - GIF
  - sticker
  - all media
- reaction bans
- shadow delete `/sdel`

Critical stealth requirements:

- muted user must see their own fake post
- fake post must preserve reply behavior when possible
- fake post must not be broadcast to others
- no explicit error should reveal the mute
- command restrictions should delete or silently ignore

Fixed:

- shadow reject reply path no longer treats `post_num` as Telegram `message_id`.
- all audited `process_shadow_reject()` call sites pass `stream`, so multilingual boards do not fall back to a Russian fake header.
- multi-reply shadow and normal paths now pass `stream` explicitly.
- `process_shadow_reject()` keeps a small per `(board_id, user_id)` fake counter, reducing duplicate fake post numbers during a burst of shadowed messages.

Remaining audit risk:

- commands and special flows call `process_shadow_reject()` from many places. Any future call must pass `reply_to_post`, not `reply_info`.
- `/sdel` depends on `PostCopies`; it cannot shadow-delete old posts whose copies were already purged.
- Fake post numbering is still not full per-recipient number virtualization. If a shadow-muted user posts many fake messages while the public counter barely moves, fake numbers can drift ahead of later real posts. A perfect version would translate all post numbers for that one recipient, which is a larger delivery-layer feature.

Implemented mitigation:
